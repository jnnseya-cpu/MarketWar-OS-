# 05 — BitriPay Payment Gateway Door & Third-Party Connector Ecosystem

---

## Part A — BitriPay Integration Layer

BitriPay is the OS-native payment gateway: a dedicated **API door** through which
merchants, partners and external platforms accept payments, settle funds, and feed
settlement-grade truth back into attribution. Architecture follows the proven
Stripe/Adyen gateway patterns: idempotent APIs, signed webhooks, sandbox parity,
ledgered money movement.

### A.1 Components

```
┌────────────────────────────── BitriPay Gateway Layer ─────────────────────────────┐
│  Merchant Portal (M-28)   Developer Centre     Revenue Management                 │
│  onboarding · KYC/KYB     API keys · sandbox   fees · revenue share · settlement  │
│  ───────────────────────────────────────────────────────────────────────────────  │
│  Payment API  ·  Payment Links  ·  QR Service  ·  Wallet  ·  Webhook Engine       │
│  ───────────────────────────────────────────────────────────────────────────────  │
│  Risk & Fraud scoring (Fraud Agent)  ·  Double-entry Ledger  ·  Settlement Engine │
└───────────────────────────────────────────────────────────────────────────────────┘
```

### A.2 Merchant onboarding flow

1. Create merchant (`POST /v1/merchants`) → status `pending_kyc`.
2. KYC/KYB via connector (Sumsub/Persona) — AML screening (ComplyAdvantage) —
   AML/KYC Agent pre-screens, human review queue for ambers.
3. Approval → sandbox keys issued automatically; live keys gated on
   `kyc=approved ∧ webhook_verified ∧ test_payment_completed`.
4. Settlement schedule chosen (T+1/T+3/weekly) and payout method verified
   (penny-drop / open-banking verification).

### A.3 Payment services

| Service | Description | Endpoint family |
|---|---|---|
| Card payments | PCI-DSS SAQ-A via hosted fields/checkout; 3DS2 | `/v1/payments` |
| QR payments | Dynamic QR per order; static QR per till | `/v1/qr` |
| Payment links | No-code link with amount/reference/expiry | `/v1/payment_links` |
| Wallet payments | OS wallet balance (M-20) spend + top-up | `/v1/wallet` |
| Bank transfers | Open-banking A2A (UK/EU), mobile money (target markets) | `/v1/bank_payments` |
| Refunds | Full/partial, policy-capped for L1 agent autonomy | `/v1/refunds` |
| Disputes | Evidence upload, timeline, outcomes | `/v1/disputes` |
| Commission split | Multi-party split at capture (marketplace M-17, affiliates M-18, agency M-16) | `split[]` on payment |
| Settlement | Batch payouts per schedule; statement API | `/v1/settlements` |

### A.4 API conventions (full spec in `07-database-and-api.md`)

- **Auth:** `Authorization: Bearer bp_live_…` / `bp_test_…`; scoped keys
  (`payments:write`, `refunds:write`, `settlements:read`…), IP allowlists optional.
- **Idempotency:** `Idempotency-Key` header honoured 24 h on all POSTs.
- **Webhooks:** HMAC-SHA256 signature header `BitriPay-Signature: t=…,v1=…`
  (timestamped, 5-min tolerance), retries with exponential backoff 24 h, replay
  from Developer Centre. Events: `payment.succeeded|failed|refunded`,
  `dispute.opened|won|lost`, `settlement.paid`, `merchant.kyc.updated`.
- **Sandbox:** full-fidelity environment; deterministic test cards/outcomes; magic
  amounts trigger declines/disputes (Stripe pattern).
- **Errors:** RFC-7807 problem+json; stable machine codes (`card_declined`,
  `insufficient_funds`, `kyc_required`, `rate_limited`).

### A.5 Revenue management

- Fee engine: per-merchant fee schedules (percentage + fixed, per method), volume
  tiers, minimums. Fees post to the ledger at capture, never at settlement
  (auditability).
- Revenue sharing: platform take vs partner share vs agency share computed by the
  split engine; every split leg is a ledger posting.
- The **OS advantage:** `payment.succeeded` events join to campaign/thread ids →
  Revenue Intelligence attributes to *cleared money*, and Budget Protection kill
  criteria can use realised margin, not proxy conversions.

### A.6 Plugin-ready integration logic

Drop-in modules shipped from the Developer Centre: JS SDK (`@bitripay/js`),
server SDKs (Node/Python/PHP), WooCommerce/Shopify plugins, WordPress shortcode,
payment-link QR posters (print asset generated by Content Factory).

## Part B — Third-Party Connector Ecosystem

Uniform connector contract (every integration implements it):

```
connect() → OAuth/keys → healthcheck() → sync(cursor) → webhooks?
scopes[] · rate_budget · data_in[] · data_out[] · consent_class · kill_switch
```

Connectors run in an isolated worker pool; failures degrade features, never the core.

### B.1 Connector governance
- Secrets in the platform secret manager, never tenant-visible; rotation SLAs.
- Each connector has a **consent class** (marketing / transactional / identity /
  financial) checked against the tenant's consent ledger at every use.
- Per-connector kill-switch in the Admin Super Control Centre (M-30).
- Rate budgets enforced platform-side so one tenant cannot exhaust a shared quota.
